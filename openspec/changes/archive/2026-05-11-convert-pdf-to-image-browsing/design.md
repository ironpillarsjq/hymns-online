## 背景

当前应用使用 PDF.js 在 `<canvas>` 元素上渲染 PDF 文档，需要：
- 约 300KB+ 的 `pdfjs-dist` 依赖及独立的 Web Worker 文件
- 复杂的 Canvas 渲染逻辑，含缩放计算和渲染任务管理
- 通过离屏 Canvas 生成缩略图，内存占用较高

迁移到基于图片的浏览后，完全移除 PDF.js，改用浏览器原生 `<img>` 渲染。这大幅简化了代码库，同时为预渲染内容提供更好的视觉保真度。

现有架构遵循规范驱动模式，共有 11 个能力规范。本次变更替换其中 2 个规范（`pdf-data-source`、`pdf-viewer`），修改另外 2 个（`page-preview-bar`、`toc-browsing`）。其余 7 个规范与数据格式无关，无需修改。

## 目标 / 非目标

**目标：**
- 用原生 `<img>` 渲染替代 PDF.js 进行所有页面展示
- 数据源从扁平的 PDF 文件变更为包含编号图片的文件夹
- 每个文件夹内的图片按数字文件名排序（如 `1.jpg` < `2.jpg` < `10.jpg`）
- 更新清单插件，发现图片文件夹而非 PDF 文件
- 更新目录页，展示文件夹（每个代表一组图片）及图片数量
- 简化预览栏，使用实际图片 URL 配合 CSS 缩放，替代 Canvas 生成的缩略图
- 保持所有现有导航行为不变（键盘、触控、点击、滚动、页码指示器、全屏、退出提示）

**非目标：**
- 不提供图片缩放/平移功能（与当前 PDF 阅读器一致）
- 不在构建时进行图片预处理或格式转换
- 不做服务端改动（保持全静态，兼容 GitHub Pages）
- 不修改色彩主题、布局切换、UI 样式规范
- 不支持混合内容（同一部署中同时包含 PDF 和图片）

## 决策

### 决策 1：使用原生 `<img>` 元素代替 Canvas

**决定**：使用标准 `<img>` 元素配合 CSS `object-fit: contain` 渲染每张图片，填满视口并保持宽高比。

**理由**：原生图片由浏览器 GPU 加速，自动处理解码，渲染无需任何 JavaScript。这消除了 PdfRenderer 中约 200 行的缩放计算、渲染任务管理和窗口调整处理代码。

**备选方案：**
- Canvas 渲染（`drawImage`）：不必要的复杂度；原生 `<img>` 用更少代码实现相同视觉效果。
- CSS background-image：过渡和预加载灵活性较差。

### 决策 2：渐进式图片预加载（当前 ± 1）

**决定**：立即加载当前图片，然后在后台使用 `new Image()` 预加载下一张和上一张图片。这样既控制了内存占用，又保证了流畅翻页。

**理由**：一次性加载所有图片对于大型图集（如 500+ 张）会消耗过多内存。只加载当前图片会导致翻页时出现明显闪烁。预加载相邻图片是最佳平衡点。

**备选方案：**
- 全量加载：对大型图集内存开销过大。
- 逐张懒加载：翻页之间有可见加载延迟，用户体验差。
- 池化预加载：对此场景过度设计；文件夹结构意味着用户通常按顺序浏览。

### 决策 3：清单结构 —— 文件夹含图片列表

**决定**：清单 JSON 从扁平的 PDF 列表变为以文件夹为中心的结构：

```json
{
  "siteTitle": "Poems",
  "folders": [
    {
      "folderName": "spring",
      "displayName": "Spring",
      "path": "data/Poems/spring",
      "imageCount": 12,
      "images": ["01.jpg", "02.jpg", "03.jpg", ...]
    }
  ]
}
```

**理由**：每个文件夹是一个独立的图片集合。`images` 数组在构建时已按数字排序，客户端无需再次排序。`imageCount` 字段使目录页无需加载所有图片即可显示数量。

**备选方案：**
- 扁平图片列表加 `group` 字段：失去文件夹作为容器的概念，用户难以组织。
- 每个文件夹单独一个清单：需要多次 HTTP 请求；对静态托管来说，单一清单更简洁。

### 决策 4：构建时按数字文件名排序

**决定**：提取文件名开头的所有数字进行数值比较来排序。例如：`"2.jpg"` < `"10.jpg"`（而非字典序的 `"10.jpg"` < `"2.jpg"`）。不含前导数字的文件按字母序排到末尾。

**理由**：用户用数字命名文件来表示顺序。字典序会产生 `1.jpg, 10.jpg, 2.jpg` 这种反直觉的结果。数值排序产生预期的 `1.jpg, 2.jpg, 10.jpg`。

**备选方案：**
- 字典序排序：对编号文件结果错误。
- 整个文件名解析为数字：对 `01-cover.jpg` 这类文件会出错。
- 强制要求零填充文件名：对用户不必要的约束。

### 决策 5：通过 CSS 缩放的 `<img>` 生成缩略图（不用 Canvas）

**决定**：预览栏将每个缩略图渲染为使用相同源 URL 的标准 `<img>` 元素，CSS 限制宽度 120px，配合 `object-fit: cover`。无需离屏 Canvas 渲染。

**理由**：这消除了约 40 行的 Canvas 缩略图生成代码和对 PDF.js `getPage()` 的依赖。浏览器内置的图片解码和缩放比 Canvas `toDataURL()` 更高效。

**备选方案：**
- 构建时预生成缩略图：增加构建复杂度；现代浏览器能高效缩放图片，无需此步骤。
- Canvas 缩略图：PDF 时代的遗留方案；对实际图片来说没有必要。

### 决策 6：组件重命名策略

**决定**：`PdfRenderer.vue` → `ImageRenderer.vue`，`PdfViewerPage.vue` → `ViewerPage.vue`。Vite 插件文件从 `vite-plugin-pdf-manifest.js` 重命名为 `vite-plugin-image-manifest.js`。

**理由**：迁移后旧名称会误导。清晰的命名提高代码可读性。旧文件直接删除；由于这是一个 BREAKING 变更，无需向后兼容兼容层。

### 决策 7：路由参数变更

**决定**：阅读器路由查询参数从 `?path=data/Poems/song.pdf` 变为 `?folder=data/Poems/spring`。阅读器页面读取文件夹路径，获取清单，从该文件夹的 `images` 数组确定所有图片 URL。

**理由**：一个文件夹包含多张图片，路由需要引用文件夹而非单个文件。阅读器从图片数组位置确定当前"页"的索引。

## 风险 / 取舍

- **[风险] 大尺寸图片在慢速连接下加载缓慢** → 缓解：渐进式预加载（决策 2）在后台加载相邻图片；首张图片能快速显示，因为阅读器从索引 0 开始。超大型源图片的用户应在放入 `/public/data/` 前进行优化。
- **[风险] 各浏览器图片格式支持不同** → 缓解：使用标准格式（JPEG、PNG、WebP），这些格式被所有浏览器普遍支持。`<img>` 元素原生处理格式协商。
- **[风险] 大量高分辨率图片占用内存** → 缓解：同时仅 3 张图片在内存中（当前 + 相邻）。超出范围的图片由浏览器垃圾回收。
- **[风险] 图片没有"页"的概念** → 缓解：每张图片视为一"页"。总"页数"等于文件夹中的图片数量。所有现有的基于页的导航（页码指示器、上一页/下一页、跳转）行为完全一致。

## 迁移计划

1. 从 `package.json` 依赖中移除 `pdfjs-dist`
2. 删除 `PdfRenderer.vue` 和 `vite-plugin-pdf-manifest.js`
3. 创建 `ImageRenderer.vue`，使用原生 `<img>` 渲染
4. 重写清单插件为 `vite-plugin-image-manifest.js`
5. 更新 `vite.config.js` 使用新插件
6. `PdfViewerPage.vue` → `ViewerPage.vue`，更新导入和基于文件夹的数据流
7. 更新 `TocPage.vue` 展示文件夹而非 PDF
8. 更新 `PagePreviewBar.vue` 使用图片 URL 而非 PDF.js 缩略图
9. 更新 `router/index.js` 中路由名称/导入的变更
10. 用户迁移数据：将 `public/data/<site>/*.pdf` 替换为 `public/data/<site>/<folder>/*.jpg`
11. 运行 `npm run build` 验证静态输出正确

**回滚：** 由于这是 BREAKING 变更且数据格式不兼容，回滚需要将代码库还原到上一个提交并恢复 PDF 文件。单一应用无需渐进式迁移路径。
